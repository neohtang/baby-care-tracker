## 🔧 Bug 修复

### 1. 修复 `wx.cloud` 初始化失败问题

**问题现象**：小程序启动后 Console 日志显示 `[SyncService] wx.cloud 尚未初始化`，云同步功能始终处于离线模式。

**根因分析**：`app.ts` 的 `onLaunch` 中 `wx.cloud.init()` 成功后立即调用 `syncService.initCloud()`，而 `syncService.initCloud()` 内部通过 `getApp().globalData.cloudReady` 间接判断云是否就绪。在 `onLaunch` 执行期间 `getApp()` 返回的实例 `globalData` 存在时序问题，导致 `cloudReady` 始终读取为 `false`。

**修复方案**：

- **`miniprogram/app.ts`**：
  - 将云开发初始化逻辑重构为 `_initCloudAndSync()` 方法
  - 增加 `wx.cloud` 存在性诊断日志（`typeof wx.cloud`）
  - 增加重试机制：`wx.cloud` 首次检测不到时，以 1 秒间隔自动重试最多 3 次
  - 第三次重试仍失败时打印详细排查清单（配置项、开发者工具状态等）

- **`miniprogram/services/sync.ts`**：
  - `initCloud()` 方法签名改为 `initCloud(envId?: string)`
  - 不再依赖 `getApp().globalData.cloudReady` 间接判断，改为**直接检测 `wx.cloud` 是否存在**
  - 支持由 `app.ts` 直接传入 `envId` 参数，绕过 `getApp()` 时序陷阱
  - 保留 `getApp()` 作为降级获取 `envId` 的兜底路径

**改动文件**：
| 文件 | 改动类型 | 说明 |
|------|---------|------|
| `miniprogram/app.ts` | 重构 | 新增 `_initCloudAndSync()` 方法，带重试和诊断日志 |
| `miniprogram/services/sync.ts` | 修复 | `initCloud()` 去除 `getApp()` 依赖，改为直接检测 + 参数传入 |

## 📋 技术细节

### 云开发初始化链路（修复后）

```
App.onLaunch()
  ↓
console.log('[App] wx.cloud 检测:', typeof wx.cloud, !!wx.cloud)
  ↓
_initCloudAndSync(retryCount = 0)
  ├── wx.cloud 不存在？→ 1秒后重试（最多3次）
  └── wx.cloud 存在
        ↓
      wx.cloud.init({ env: CLOUD_ENV_ID, traceUser: true })
        ↓
      this.globalData.cloudReady = true
        ↓
      syncService.initCloud(CLOUD_ENV_ID)  ← 直接传入 envId
        ├── 直接检测 wx.cloud（不依赖 getApp()）
        ├── 设置 isCloudReady = true
        └── return true
              ↓
            callFunction('getOpenId') → 缓存 openid
              ↓
            syncService.startAutoSync()
            syncService.startWatch()
```

## 🔍 已知问题与注意事项

1. **云函数需手动部署**：`getOpenId` 和 `hunyuan-chat` 两个云函数需要在微信开发者工具中右键「上传并部署：云端安装依赖」
2. **环境 ID 硬编码**：当前云开发环境 ID `neo3-7gtg0bdtc9fcc672` 硬编码在 `app.ts` 中，后续可考虑移至配置文件或环境变量
3. **`getApp()` 时序陷阱**：此次修复揭示了微信小程序 `onLaunch` 期间 `getApp()` 的经典时序问题，后续其他服务如需在启动阶段读取 `globalData`，建议采用参数直传而非 `getApp()` 间接获取
